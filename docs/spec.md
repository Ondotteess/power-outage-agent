# Power Outage Agent (MVP)

## 1. Обзор

Power Outage Agent — система, которая регулярно собирает данные о плановых отключениях электроэнергии, нормализует их и сопоставляет с офисами компании для генерации уведомлений о потенциальном влиянии.

Источники данных:

- сайты энергокомпаний (основной источник)
- Telegram-каналы администраций и коммунальных служб
- новостные ресурсы
- официальные уведомления городских/региональных органов

## 2. Основной сценарий

1. Scheduler по расписанию ставит задачи в очередь
2. Collectors забирают сырой контент из источников
3. Adaptive Parser извлекает структурированные данные (regex/HTML-селекторы или LLM-fallback)
4. Сырой контент сохраняется в Raw Storage (артефакт для аудита и повторного парсинга)
5. LLM-нормализатор приводит данные к единому формату → Clean Storage
6. Dedup-слой убирает дубликаты
7. Office Matcher сопоставляет отключение с офисами компании
8. При совпадении публикуется событие `OfficeOutageRiskDetected` в шину
9. Notifier рассылает уведомление: дашборд и корпоративные каналы 
10. Admin UI показывает dashboard, matcher output, notifications и карту офисов `/map`

## 3. Логическая архитектура

### 3.1 Scheduler

- Тригерит pipeline по расписанию
- Создаёт задачи `fetch_source` для каждого активного источника
- Не знает о парсинге, нормализации и матчинге — только о реестре источников и их частоте опроса
- Частота опроса задаётся per-source (например, сайт энергокомпании — раз в 6 ч, телеграм-канал админисрации — раз в 30 мин)

### 3.2 Task System

Очередь задач со следующими типами:

- `fetch_source` — забрать сырой контент
- `parse_content` — извлечь структурированные записи
- `normalize_event` — привести к единому формату через LLM
- `deduplicate_event` — проверить на дубликат и слить с существующим
- `match_offices` — сопоставить с офисами
- `emit_event` — опубликовать в шину

Требования:

- retry с экспоненциальным backoff (см. §6.2)
- идемпотентность по `task_id` + хешу входных данных
- DLQ для задач, упавших после исчерпания retry

### 3.3 Collectors

Отвечают **только** за получение данных. Не интерпретируют контент.

Поддерживаемые типы источников:

- HTML страницы
- RSS / JSON endpoints
- Telegram API (через bot/userbot)
- произвольные новостные страницы

На выходе — сырой контент + метаданные источника. Никакой бизнес-логики.

### 3.4 Adaptive Parser

Парсер выбирает стратегию на основе типа источника и его профиля в реестре:

1. **Структурированный путь** — для источников с известной разметкой (конкретный сайт энергокомпании). HTML-селекторы / regex / JSON-path. Дёшево, быстро, детерминировано.
2. **LLM-путь (fallback)** — когда структурированный парсер не сработал или источник не имеет фиксированной разметки (телеграм-каналы, новости). Промпт извлекает поля по схеме нормализованного события.

Выбор стратегии зашит в профиль источника, а не определяется на лету. 

### 3.5 Raw Storage

Сохраняет исходные ответы источников as-is. Используется для:

- аудита (что именно мы видели)
- дебага парсера
- повторного парсинга при изменении логики экстракции

Схема записи — см. §7.1.

### 3.6 Normalization Layer (LLM)

Приводит `ParsedRecord` к единой схеме `NormalizedEvent` (см. §7.2). Текущая реализация использует Sber GigaChat напрямую через `GigaChatClient`. Поля `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL` оставлены как OpenAI-compatible baseline для будущего мульти-провайдера, но сейчас активный `LLMNormalizer` их не использует.

Помимо извлечения полей:

- классифицирует тип события (`power_outage` / `maintenance` / `infrastructure_failure` / `other`)
- проставляет `confidence` (используется в дедупе и матчинге как тай-брейкер)
- нормализует временные метки в UTC
- нормализует адреса до уровня "город + улица + дом" (где возможно)
- не выдумывает отсутствующие номера домов; при неполном адресе снижает `confidence`

Для защиты от массовых LLM-вызовов на больших источниках профиль источника поддерживает:

- `normalize_enabled=false` — парсер сохраняет `ParsedRecord`, но не ставит `normalize_event`;
- `normalize_limit=N` — ставит не более `N` задач нормализации на один raw-ответ.

### 3.7 Dedup Engine

Убирает дубликаты по композитному ключу: **нормализованный адрес + временной интервал** (с допуском ±15 мин на границах).

Стратегии:

1. Точное совпадение ключа → merge, источники складываются в массив
2. Частичное совпадение (один и тот же адрес + пересекающиеся интервалы) → merge с расширением интервала
3. Конфликт данных (разные причины, разные `end_time`) → побеждает запись с большим `confidence`; проигравшая версия остаётся в истории события

### 3.8 Office Matcher

Сопоставляет нормализованные события с реестром офисов.

- Реестр офисов — в закрытом контуре. Доступ через внутренний API/выгрузку (TBD на неделе 3)
- Адреса офисов проходят ту же нормализацию, что и адреса событий
- MVP: точное совпадение по нормализованному адресу
- Расширение: гео-радиус, сопоставление по подстанции/фидеру (если источник даёт)

Выход — см. §7.3.

### 3.9 Notifier

Подписан на события `OfficeOutageRiskDetected` из шины. Каналы доставки:

- лог + дашборд (всегда)
- Telegram

### 3.10 Admin UI и карта офисов

Admin UI — отдельный Vite/React интерфейс поверх FastAPI `/api/*`.

Страница `/map` показывает офисы на настоящей карте:

- зелёный маркер: `ok`
- жёлтый маркер: `risk`
- красный маркер: `critical`

Источник данных — `GET /api/map/offices`. Endpoint отдаёт готовую UI-проекцию:
офис, адрес, координаты, текущий статус и активные impacts.

Координаты офисов задаются вручную в БД/seed:

- `latitude`
- `longitude`

Автоматический геокодинг запрещён для MVP: не подключаем Google/Yandex Maps API,
не требуем ключи, не добавляем лимиты и платный внешний контур. Офис без координат
остаётся в ответе API, но не получает маркер; frontend показывает его в списке
`Missing coordinates`.

## 4. Архитектурные принципы

Система проектируется как минимальный, но расширяемый event-driven pipeline. Фокус MVP — **стабильность извлечения, нормализации и матчинга**, а не универсальная платформа.

### 4.1 Принципы

- **Адаптивная обработка** источников: добавление нового источника = новая запись в реестре + (опционально) новый профиль парсера. Никаких изменений в pipeline
- **Устойчивость к грязным данным**: каждый шаг должен корректно обрабатывать пустые/частичные/битые поля и не валить весь pipeline
- **Сохраняем сырое**: никогда не выбрасываем raw-данные, чтобы можно было перепарсить
- **Идемпотентность**: повторный запуск любого шага с теми же входами даёт тот же результат и не создаёт дубль-эффектов

### 4.2 Шина событий

События pipeline:

- `RawFetched` — collector сохранил raw
- `RawParsed` — парсер извлёк структурированные данные
- `EventNormalized` — данные приведены к единой схеме
- `EventDeduplicated` — выполнен dedup, событие готово к матчингу
- `OfficeImpactDetected` — найден офис под удар
- `NotificationEmitted` — уведомление отправлено

Внутренние события (`Raw*`, `Event*`) нужны прежде всего для трейсинга. Внешний потребитель (Notifier, дашборд) подписывается только на `OfficeImpactDetected`.

## 5. Ограничения MVP

- 3 сайта энергокомпаний
- 1–2 альтернативных источника (Telegram/новости)
- 1–2 региона присутствия офисов
- Только плановые отключения
- Уведомления — в Telegram + дашборд;

## 6. Соглашения

Раздел собирает кросс-компонентные правила, чтобы не дублировать их по тексту.

### 6.1 Форматы данных

- **Время**: всё хранится и передаётся в **UTC, ISO 8601** (`2025-11-12T08:30:00Z`). Отображение в локальном TZ — на стороне Notifier/дашборда
- **Идентификаторы**: UUIDv4, в snake_case полях (`event_id`, `office_id`, `task_id`)
- **Адреса**: нормализованная форма — `"<город>, <улица>, <дом>[, <корпус>]"`, нижний регистр, без сокращений (`улица`, не `ул.`)
- **JSON**: snake_case, без вложенности глубже 3 уровней без явной необходимости
- **Severity / impact_level**: core enum `low | medium | high`; map/API projection допускает `unknown` и `critical` как UI-уровни статуса угрозы

### 6.2 Retry и идемпотентность

- Backoff: экспоненциальный, base=2s, max=10 мин, max attempts=5
- Идемпотентность задач — по `(task_type, input_hash)`
- Идемпотентность событий шины — по `event_id`; повторная публикация не порождает повторных уведомлений

### 6.3 Логирование и трейсинг

- Каждая запись в pipeline имеет `trace_id`, который протаскивается через все события и задачи
- Любая запись в Raw Storage привязана к `trace_id` источника

## 7. Схемы данных

### 7.1 Raw Storage Record

```json
{
  "id": "uuid",
  "source_id": "uuid",
  "source_url": "string",
  "source_type": "html | telegram | rss | json | other",
  "raw_content": "string",
  "content_hash": "sha256",
  "fetched_at": "iso8601",
  "trace_id": "uuid"
}
```

### 7.2 Normalized Event

```json
{
  "event_id": "uuid",
  "parsed_record_id": "uuid | null",
  "event_type": "power_outage | maintenance | infrastructure_failure | other",
  "start_time": "iso8601",
  "end_time": "iso8601 | null",
  "location": {
    "raw": "string",
    "normalized": "string",
    "city": "string",
    "street": "string",
    "building": "string"
  },
  "reason": "string | null",
  "sources": ["raw_record_id", "..."],
  "confidence": 0.0
}
```

### 7.3 Office

```json
{
  "id": "uuid",
  "name": "string",
  "city": "string",
  "address": "string",
  "region": "string",
  "is_active": true,
  "latitude": 56.4846,
  "longitude": 84.9476,
  "extra": {}
}
```

`latitude` / `longitude` nullable. Координаты задаются вручную. Отсутствие координат
не блокирует matcher и уведомления.

### 7.4 Office Impact

```json
{
  "office_id": "uuid",
  "event_id": "uuid",
  "impact_start": "iso8601",
  "impact_end": "iso8601 | null",
  "impact_level": "low | medium | high",
  "match_strategy": "exact_address | geo_radius | feeder",
  "detected_at": "iso8601"
}
```

### 7.5 Map Offices API Response

`GET /api/map/offices`:

```json
{
  "offices": [
    {
      "id": "uuid",
      "name": "string",
      "address": "string",
      "city": "string",
      "region": "string",
      "latitude": 56.4846,
      "longitude": 84.9476,
      "status": "ok | risk | critical",
      "active_impacts": [
        {
          "id": "uuid",
          "reason": "string | null",
          "severity": "low | medium | high | critical | unknown",
          "starts_at": "iso8601",
          "ends_at": "iso8601 | null",
          "event_type": "power_outage | maintenance | infrastructure_failure | other | null"
        }
      ]
    }
  ]
}
```

Статус:

- `ok`: нет активных impacts
- `risk`: активный low/medium/unknown impact
- `critical`: active high/critical impact или событие явно означает outage/closure

Активный impact: `starts_at <= now` и `ends_at` отсутствует или ещё не наступил.

### 7.6 Notification

```json
{
  "notification_id": "uuid",
  "office_id": "uuid",
  "event_id": "uuid",
  "type": "power_outage | disruption",
  "severity": "low | medium | high",
  "start_time": "iso8601",
  "end_time": "iso8601 | null",
  "source_summary": "string",
  "channels": ["dashboard", "telegram", "..."],
  "emitted_at": "iso8601"
}
```

## 8. Открытые вопросы

- Доступ к реестру офисов (закрытый контур) — формат и периодичность синхронизации
- Промпты нормализатора — нужен эталонный набор на 20–30 реальных объявлений для оценки качества
- Стратегия дедупа при противоречивых источниках — нужен ли human-in-the-loop на этапе MVP или достаточно `confidence`-тай-брейкера
- Метрики качества: precision/recall по матчингу, latency от появления в источнике до уведомления
- Синхронизация координат офисов: пока вручную, позже нужен внутренний источник правды без внешнего геокодинга
